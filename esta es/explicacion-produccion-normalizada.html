¿Dónde están leche, matadero y sacrificio?

Explicación del módulo de producción — base de datos bd_crud normalizada (3FN)

Los datos no desaparecieron. Ya no viven en tres tablas anchas llamadas leche, matadero_vivo y sacrificio_vacuno, sino en un modelo normalizado en filas (entidad–atributo–valor).

1. Antes (esquema original)

Cada módulo era una tabla con muchísimas columnas y una fila por fecha:

Tabla antigua Qué guardaba
sacrificio_vacuno Un día de sacrificio + columnas como terneras_Cab, vacas_Cbz_1ra, total1_Cab, etc.
matadero_vivo Lo mismo, pero con datos de matadero vivo.
leche Un día de leche + columnas como Zenea_Acopio, Rosafe_TOTAL, total1_Acopio, etc.

Ese diseño concentraba cientos de columnas en una sola tabla por módulo. Los campos total1_*, total2_*, etc. repetían información que se puede calcular a partir de otros valores.

2. Ahora (dónde está “la pelota” de datos)

Las tres áreas de negocio siguen existiendo, pero como módulos en prod_modulo, no como tablas físicas separadas:

id_modulo codigo nombre
1sacrificioSacrificio vacuno
2mataderoMatadero vivo
3lecheLeche

Los números se reparten en cuatro tablas relacionadas:

prod_modulo
sacrificio / matadero / leche → prod_registro
1 fila = módulo + fecha → prod_valor
valor numérico ← prod_metrica
nombre de columna
Tabla Rol Equivalente mental
prod_modulo Catálogo de los 3 tipos “¿Es leche, matadero o sacrificio?”
prod_registro id_modulo + fecha + auditoría La fila del día (antes el encabezado de la tabla vieja)
prod_metrica clave = nombre de cada columna El diccionario de columnas (Nazareno_Acopio, aniojas_Cab, …)
prod_valor (id_registro, id_metrica, valor) Cada celda numérica de la planilla

Ejemplo concreto

  • Antes: en leche, fila fecha = 2024-03-15, columna Zenea_Acopio = 120.
  • Ahora:
    • prod_registro: módulo leche (id 3), fecha 2024-03-15 → un id_registro.
    • prod_metrica: clave Zenea_Acopio → un id_metrica.
    • prod_valor: ese id_registro + ese id_metrica → valor = 120.

La migración desde el dump original hizo exactamente esa conversión:

migrateProd('sacrificio_vacuno', 'sacrificio', 1);
migrateProd('matadero_vivo',   'matadero',   2);
migrateProd('leche',           'leche',      3);

3. ¿Y los totales (total1_, total2_, …)?

No se guardan en la base de datos. Se calculan al leer en el servidor (igual que en el frontend), a partir de las métricas “reales”. Solo se persisten las claves que no son categorías totalizadoras (total1, total2, total3, total, etc.).

4. ¿El frontend sigue viendo lo mismo?

Sí. Las rutas de la API no cambiaron de nombre para el usuario:

  • /leche, /create-leche, …
  • /matadero, …
  • /sacrificio, …

El backend, al guardar, descompone el objeto ancho en prod_registro + prod_valor. Al listar, vuelve a armar el mismo JSON con todas las claves (incluidos totales calculados). Para quien usa la aplicación, la planilla es la misma; solo cambió la forma de persistir en MySQL.

5. Histórico de cambios

Antes Ahora
produccion_historico con JSON en datos_json historico_produccion (cabecera: módulo, fecha, acción, usuario)
+ historico_produccion_valor (mismas métricas que prod_valor, fila a fila)

El modal del historial en el frontend sigue recibiendo un objeto datos reconstruido por el backend.

Resumen en una frase

leche, matadero_vivo y sacrificio_vacuno no se borraron como conceptos: pasaron a ser id_modulo 1, 2 y 3 en prod_modulo, y cada columna numérica de esas tablas antiguas es ahora una o más filas en prod_valor, enlazadas por prod_metrica.
