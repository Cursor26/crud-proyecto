Usuarios, contratación y RRHH

Cómo quedó cada módulo en bd_crud normalizada — complemento del documento de producción

Estos tres bloques no desaparecieron: se reorganizaron para cumplir 3FN (sin repetir textos de catálogo en cada fila, sin claves “sueltas” como id_tabla, sin columnas calculadas guardadas en disco). El frontend sigue hablando casi igual; el backend traduce entre la forma antigua y las tablas nuevas.

1. Usuarios y seguridad

Antes

La tabla usuarios guardaba el rol como texto en la columna rol (admin, rrhh, contratacion, …). Cada fila repetía el mismo literal; si cambiabas el nombre del perfil, había que tocar muchas filas.

Ahora — 3 tablas

roles
catálogo de perfiles 1 → N usuarios
cuentas de acceso 1 → N password_reset_tokens
recuperar contraseña
TablaQué guardaClave
roles Perfiles del sistema: admin, rrhh, contratacion, produccion, estadistica, director PK id_rol · codigo único
usuarios Email, nombre, contraseña (bcrypt), si está activo, auditoría PK email · FK id_rol → roles
password_reset_tokens Tokens temporales para “olvidé mi contraseña” (hash, vencimiento, IP) PK id · email indexado (sin FK obligatoria)

Ejemplo mental

  • Antes: usuarios.rol = 'rrhh' (texto en cada usuario).
  • Ahora: fila en roles con codigo = 'rrhh' → usuarios.id_rol = 2.

¿El login y la app ven lo mismo?

Sí. Al autenticar o listar usuarios, el backend hace JOIN y devuelve rol como código de texto (igual que antes):

SELECT … r.codigo AS rol FROM usuarios u INNER JOIN roles r ON r.id_rol = u.id_rol

Al crear o editar un usuario, envías rol: "rrhh" en el JSON; el servidor lo convierte a id_rol internamente.

Resumen usuarios: el concepto “rol” pasó de columna repetida a tabla catálogo roles; los datos de cuenta siguen en usuarios.

2. Contratación

Antes

En contratos_generales convivían varios problemas de normalización:

  • proveedor_cliente como número suelto (0 = cliente, 1 = proveedor) sin tabla de dominio.
  • tipo_contrato como texto libre repetido en cada contrato (“Compra”, “Servicio”, …).
  • vencido guardado en BD aunque depende solo de fecha_fin y la fecha de hoy.

Ahora — catálogos + contrato

catalogo_tipo_contraparte → contratos_generales ← catalogo_tipo_contrato
TablaQué guardaNotas
catalogo_tipo_contraparte Cliente (código 0) y Proveedor (código 1), compatibles con el valor antiguo Evita repetir el significado del 0/1 en cada fila
catalogo_tipo_contrato Tipos únicos: Compra, Servicio, Arrendamiento, … (se crean al migrar según los textos que ya existían) Un nombre → un id_tipo_contrato
contratos_generales Número de contrato, empresa, correo, suplementos, vigencia, fechas PK numero_contrato · FK id_contraparte, id_tipo_contrato

Campos que ya no están en la tabla física

En la API / pantalla (igual que antes)En MySQL ahora
proveedor_cliente (0 o 1) id_contraparte → el backend expone otra vez proveedor_cliente con JOIN al catálogo
tipo_contrato (texto) id_tipo_contrato → el SELECT devuelve el nombre del catálogo como tipo_contrato
vencido (0/1) Calculado en consulta: fecha_fin < CURDATE() — no se guarda

Ejemplo mental

  • Antes: contrato X con proveedor_cliente = 1, tipo_contrato = 'Compra', vencido = 1.
  • Ahora: misma fila con id_contraparte = 2 (proveedor), id_tipo_contrato = 3 (id de “Compra” en catálogo); al listar, la API arma otra vez los tres campos “viejos”.
Resumen contratación: los contratos siguen en contratos_generales; lo que cambió es que cliente/proveedor y tipo de contrato apuntan a catálogos, y vencido se calcula al vuelo.

3. Recursos Humanos

Antes — el eje era id_tabla

Muchas tablas de RRHH usaban id_tabla (a veces el carnet, a veces un identificador interno) sin FK formal a empleados. Además:

  • cargos tenía departamento en texto libre.
  • empleados mezclaba datos con textos largos duplicados (cursos, certificados en columnas).
  • segseguridad concentraba varios accidentes/incidencias en columnas de una sola fila (grupos repetitivos).

Ahora — el eje es empleados.carnet_identidad

Toda persona trabajadora vive en empleados (PK = carnet de 11 caracteres). El resto de tablas enlazan con FK carnet_identidad.

3.1 Núcleo organizacional

departamentos
jerarquía id_padre → cargos → empleados

3.2 Módulos “uno por empleado” (relación 1:1)

Cada tabla tiene UNIQUE(carnet_identidad): como máximo una fila por trabajador. Es el mismo concepto que antes tenías en pantallas separadas (asistencia, cursos, evaluaciones, etc.).

asistencias certificaciones cursos evalcapacitacion evaluaciones objetivos salarios seguridad vacaciones segseguridad

En consultas SQL del backend, muchas rutas siguen devolviendo el alias id_tabla = carnet_identidad para que el frontend no cambie.

3.3 Seguridad industrial descompuesta

Antes: varias columnas de accidentes en una fila de segseguridad. Ahora:

  • segseguridad — cabecera por empleado (carnet).
  • segseguridad_incidencia — una fila por incidencia (cantidad, descripción, orden).

Eso cumple 1FN: no hay “accidente1, accidente2, accidente3” en la misma fila.

3.5 Grupos de trabajo

grupos_trabajo N:M grupo_miembros ↔ empleados

  • asistencia_grupal: control de asistencia del grupo en una fecha.

Resumen RRHH: no hay una sola tabla gigante; hay un hub (empleados) y satélites por tema. La clave de negocio unificada es el carnet, no id_tabla suelto.

Los tres módulos en una frase

  • Usuarios: roles en catálogo; cuentas en usuarios; la API sigue enviando rol como texto.
  • Contratación: contratos en contratos_generales; tipos y cliente/proveedor en catálogos; vencido calculado.
  • RRHH: todo gira en torno a empleados.carnet_identidad; tablas por función; incidencias y grupos normalizados.

Producción (leche, matadero, sacrificio) se explica aparte porque usa otro patrón (registro + métrica + valor). Ver explicacion-produccion-normalizada.html.
